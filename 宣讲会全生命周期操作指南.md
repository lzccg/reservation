# 宣讲会全生命周期操作指南

---

## 阶段一：企业提交申请（0 - 待审）

**业务场景：**
企业在系统前端填写宣讲会的主题、时间、地点、容量等信息并提交。

**数据操作（Insert）：**

* 系统默认插入 `session_status = 0`（待审）
* **核心防错：**
  必须在插入时，将 `remaining_seats（剩余座位）` 初始化为 `capacity（场地总容量）`
* 学生端此时不可见该宣讲会

---

## 阶段二：管理员场地审核（1 - 已审核 / 5 - 已驳回）

**业务场景：**
管理员在后台查看“待办列表”，核对全校场地和时间是否有冲突。

**数据操作（Update）：**

* **审核通过：**

  * 将 `session_status` 更新为 `1`（已审核）
  * 记录当前时间到 `audit_time` 字段
  * 此时宣讲会依然不对学生开放预约

* **审核驳回：**

  * 将 `session_status` 更新为 `5`（已驳回）
  * 必须填写 `audit_remark（驳回原因）`
  * 企业端收到通知后可修改并重新提交（状态变回 `0`）

---

## 阶段三：正式面向学生发布（2 - 已发布）

**业务场景：**
到达公布时间后，由管理员或企业点击“发布”，或系统根据 `publish_time` 自动发布。

**数据操作（Update）：**

* 将 `session_status` 从 `1` 更新为 `2`（已发布）
* 只有状态为 `2` 时：

  * 学生端“宣讲会大厅”才会展示该记录
  * 学生才能进行预约操作

---

## 阶段四：学生抢座与取消（高并发阶段）

**业务场景：**
宣讲会处于“已发布”状态期间，学生进行预约或取消预约。

**数据操作（并发控制 Update）：**

* 必须带上状态条件和容量限制，防止超卖
* 利用 MySQL 行锁实现安全扣减

```sql
-- 预约扣减座位
UPDATE session
SET remaining_seats = remaining_seats - 1
WHERE session_id = ?
  AND session_status = 2  -- 必须是已发布状态
  AND remaining_seats > 0;
```

---

## 阶段五：系统自动归档（3 - 已结束）

**业务场景：**
宣讲会结束后，系统自动关闭预约与签到功能。

**数据操作（定时任务 Update）：**

* 使用 Spring Boot `@Scheduled` 定时任务定期扫描数据库
* 执行批量更新：

  * 将所有 `session_status = 2`
  * 且当前时间 > `end_time`
  * 更新为 `3`（已结束）
* 企业与管理员可下载最终签到报表

---

## 阶段六：突发异常终止（4 - 已取消）

**业务场景：**
因突发情况（天气、主讲人缺席等）取消宣讲会。

**数据操作（联级 Update）：**

* 将 `session_status` 更新为 `4`（已取消）
* 必须填写 `cancel_reason（取消原因）` 并向学生公示

**级联处理：**

* 更新 `reservation（预约记录表）`：

  * 将所有相关预约记录标记为“已取消”
* 通过系统消息通知所有已预约学生

---

## 状态流转总结

| 状态值 | 状态名称 | 说明        |
| --- | ---- | --------- |
| 0   | 待审   | 企业提交后初始状态 |
| 1   | 已审核  | 管理员审核通过   |
| 2   | 已发布  | 对学生开放预约   |
| 3   | 已结束  | 宣讲会结束归档   |
| 4   | 已取消  | 异常终止      |
| 5   | 已驳回  | 审核未通过     |

---

## 一、剩余座位的核心计算公式

从底层数据角度来看，宣讲会的真实剩余座位数遵循如下绝对公式：

```
实际剩余座位 = capacity（场地总容量） - 有效预约总数
```

**有效预约定义：**

* `reservation` 表中状态为：

  * `0（待签到）`
  * `2（已签到）`
  * `3（爽约）`
* 以上状态均占用座位

**不计入占用：**

* 状态为 `1（已取消）` 的记录（视为已释放座位）

> 注意：
> 虽然该公式是最终对账依据，但在实际系统中 **不能每次实时计算数据库**，否则会严重影响性能。

---

## 二、系统各阶段的“剩余座位”计算流程

### 1. 初始阶段（宣讲会发布时）

当管理员审核通过并将宣讲会状态变更为 `2（已发布）` 时，系统进行初始化计算：

**MySQL 操作：**

* 将 `session` 表的 `remaining_seats` 字段直接赋值为 `capacity`

**Redis 操作：**

* 将剩余容量同步写入 Redis 缓存
* 示例：设置键值 `session:seats:1001 = 200`

---

### 2. 抢座阶段（高并发扣减）

当大量学生同时点击“预约”时，计算必须在 Redis 内存中完成。

**Redis 原子扣减：**

* 使用 `DECR` 命令

**边界判断：**

* 返回值 `>= 0`：抢座成功
* 返回值 `< 0`：执行 `INCR` 回滚并提示“名额已满”

---

### 3. 落库阶段（异步或事务扣减）

```sql
UPDATE session
SET remaining_seats = remaining_seats - 1
WHERE session_id = ?
  AND remaining_seats > 0;
```

---

### 4. 取消阶段（名额释放回滚）

* MySQL：`remaining_seats + 1`
* Redis：执行 `INCR`

---

## 三、兜底校准机制（对账）

* 定时任务每 30 分钟执行一次

**核心公式：**

```
remaining_seats = capacity - 有效预约总数
```

**校准策略：**

* 若不一致：

  * 强制更新 MySQL
  * 同步更新 Redis

**目标：**

* 保证最终一致性
