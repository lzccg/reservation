01 研究背景

线下管理的痛点

  • 高校企业宣讲会场次日益增多，传统线下人工签到效率低下
  • 纸质打卡或普通二维码扫码入场拉长排队时间
  • 极易出现"代签"、"占座"以及"漏签"等不规范情况

传统信息化手段的不足

  • 普通预约系统缺乏有效的现场身份强校验机制
  • 无法确保到场人员与预约人员的真实一致性

01 研究意义与目标

业务闭环管理

企业发布活动
→
学生在线预约
→
现场刷脸打卡
→
后台数据统计

智能化升级（核心目标）

引入人工智能活体人脸识别技术，取代传统打卡。不仅大幅提升打卡准确率与效率，更从根本上保障了考勤数据的真实与安全。

为高校智慧校园的行政信息化建设提供一种可行的现代化参考方案。

03 核心创新点一：高并发抢座处理

场景难点

热门企业宣讲会发布后，瞬间会有大量学生同时发起预约请求，极易导致数据库崩溃或产生"超卖"（库存变为负数）现象。

解决方案（Redis + Lua）

  • 在学生预约环节，将宣讲会剩余座位数据同步至 Redis 缓存
  • 使用 Lua 原子脚本处理高并发下的库存扣减与用户防重判断，保障数据的一致性与操作的原子性
兜底机制：通过 Spring 事务管理（@Transactional）与数据库乐观锁机制，双重保障最终落库数据的绝对正确

03 核心创新点二：人脸识别与防代签

底库建立与特征提取

  • 学生上传正面照
  • 系统后端调用 API 审核照片质量（限制 < 5MB 的单人正面照）
  • 合格后转化为人脸特征码并存入底库（绑定学生ID）

现场 1:N 活体核验

  • 采用成熟的阿里云人脸人体 SDK（1:N 人脸搜索）接口
  • 现场设备捕捉图像发送至服务端，与底库数据比对
  • 当系统判断匹配置信度评分 > 80.0 且存在预约记录时，系统自动更新为"已签到"状态
从根本上避免了传统的纸质或二维码代签问题

04 核心研究成果 —— 宣讲会全链路调度

发布与审核规范化

企业申请发布
→
待审状态
→
管理员审批
→
发布成功
  • 企业填写场地容量、时间等信息申请发布，数据进入"待审"状态
  • 管理员拥有"强制干预"权限，可对宣讲会进行审核通过、驳回甚至在活动前/中途"强制叫停"，保障校园活动的安全可控
  • 审核通过后，活动自动在学生大厅界面展示，进入预约生命周期

04 核心研究成果 —— 学生预约与信息流转

预约机制完善

  • 学生可根据企业名称、日期范围模糊搜索宣讲会
  • 实时显示剩余名额，座位为 0 时拦截预约

防"爽约"机制

支持学生在宣讲会开始前自主取消预约，取消后座位库存自动回退至奖池（Redis 与 MySQL 均回滚）

这样既保障了企业资源利用率，也给了学生灵活调整的空间。

04 核心研究成果 —— 现场无感签到

设备协同与状态判定

  • 管理员/企业在现场开启签到摄像头模块
  • 系统严格校验签到时间窗
正常签到

在设定时间（如宣讲会开始 15 分钟内）完成人脸识别 → 判定为"正常"

迟到

超出时间范围完成签到 → 精确标记为"迟到"状态

未预约该场活动的人脸闯入时，系统予以拦截提示，保障会场秩序

04 核心研究成果 —— 数据统计与可视化

全局数据看板

  • 宣讲会总场次
  • 总预约人次
  • 实际签到人次
  • 整体签到率

一键报表导出

支持通过 Apache POI 等技术，将某场活动的最终考勤明细（包含序号、姓名、学号、学院、班级、签到状态与时间）一键导出为 Excel 报表，极大地减轻了辅导员与校企核对数据的行政负担。

05 系统测试与分析

测试方案

主要采用黑盒测试（功能测试），验证各接口输入输出是否符合规范定义

测试结果摘要

  • 覆盖了用户管理、宣讲会管理、预约模块、人脸签到模块 5 大核心场景的数十个测试用例
  • 包含异常流程测试：如无脸/多人脸上传报错拦截、名额满载限制、超时取消等均达到预期反馈
结论：系统逻辑严谨，界面响应迅速，运行稳定可靠

06 研究结论与不足展望

研究结论

本项目成功将 Spring Boot、Redis、人脸识别技术有效整合，达成预期目标，提升了校园活动组织效率，具有高度实用价值。

当前不足

  • 对人脸识别技术的理解主要停留在 SDK 接口调用层面，底层算法掌握尚不深入
  • 极端高并发下更细粒度的数据一致性处理还有优化空间

未来展望

进一步优化基础架构，横向扩展至校园学术讲座、文体比赛等其他活动场景，助力智慧校园无纸化全面普及。
