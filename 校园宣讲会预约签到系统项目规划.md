# 校园宣讲会预约签到系统项目规划

## 一、项目概述

随着高校信息化水平的不断提升，传统宣讲会预约与签到方式逐渐暴露出效率低、管理混乱、易出现代签等问题。为解决上述问题，本项目设计并开发一套基于 Spring Boot 的校园宣讲会预约签到系统。

系统通过引入在线预约与人脸识别签到技术，实现宣讲会管理的智能化与规范化，提升管理效率和数据准确性。

---

## 二、项目目标

本项目的主要目标包括：

* 实现宣讲会信息的线上发布与管理
* 提供学生在线预约宣讲会功能
* 引入人脸识别技术实现无接触签到
* 构建完整的签到数据统计与分析功能
* 提高宣讲会组织效率与管理规范性

## 五、技术方案规划

### 1. 系统架构

系统采用前后端分离架构：

* 前端：Vue + Element Plus 使用 pnpm 管理依赖，代码统一放置在 /src/main/resources/frontend 目录下
* 后端：Spring Boot + MyBatis-Plus +Java 17 + MyBatis + Maven
* 数据库：MySQL 8.0 (账号密码: root / 123456)* 缓存：Redis
* 第三方服务：人脸识别 API

---

### 2. 技术特点

* 前后端分离，提高开发效率
* 使用 RESTful API 进行通信
* 引入缓存提升系统性能
* 使用人脸识别增强安全性

## 六、数据库设计规划

系统主要设计以下核心数据表：

## 七、解决方案规划

### 系统集成与协同设计
五大子系统通过统一的业务流程和数据库关联实现生态闭环 ：

* 数据流转：用户管理模块提供身份认证 ；宣讲会模块提供预约基础数据 ；预约模块生成签到依据 ；人脸模块完成身份验证 。

* 并发控制协同：学生提交预约申请时，系统自动执行名额校验与重复校验 ，底层由Redis Lua脚本保障操作原子性。

* 前端工程协同：前后端分离部署，前端项目集成在 /src/main/resources/frontend，通过Axios与后端Spring Boot进行RESTful API通信，利用Element-Plus组件库快速构建后台，ECharts消费后端统计接口渲染图表。

### 核心图表与数据模型设计说明
1. 实体与类图设计原则 实体划分：定义 Student, Company, Admin, Session, Reservation, Checkin, FaceData 七大核心类 。关联关系（解决多对多）：通过建立 Reservation (预约记录) 和 Checkin (签到记录) 实体，将学生与宣讲会之间的多对多关系拆解为一对多，便于追踪单次履约状态 。数据封装：严格按照MVC分层，实体类配合Lombok的 @Data 注解简化代码，MyBatis-Plus处理持久层映射。
2. 业务流程与活动图规划 发布与审核流程：企业后台发布 -> 状态设为“待审核” -> 管理员校验冲突 -> 审核通过开放预约 。签到核心流程：到达现场 -> 打开页面采集图像 -> 上传服务器调用接口 -> 1:1比对 -> 判断结果（成功/失败/重复） -> 更新状态写入数据库 。

### 界面（UI）设计原则与图表展示
前端基于 Vite 构建，采用 Vue3 组合式 API 开发，UI 架构及交互设计原则如下：
* 统一风格与组件库：全量使用 Element-Plus 组件库（表格、表单、对话框、消息提示），保证后台管理系统视觉与交互的高度一致性。
* 数据可视化驱动（ECharts集成）：在管理员与企业的“工作台/仪表盘”首页，引入 ECharts。图表展示功能：渲染柱状图（各学院/专业预约人数分布）、饼图（单场宣讲会签到率/缺席率分析）、折线图（近期宣讲会热度趋势）。
* 动态路由与权限控制：基于用户登录返回的 Role 动态生成左侧菜单栏，学生仅见预约与签到，企业见宣讲发布与统计，管理员见全局审批。
* 硬件调用交互体验：在“人脸信息录入”与“现场签到”页面，调用 WebRTC API 开启本地摄像头，界面需提供清晰的取景框指引与实时的质量检测（如模糊度）文字反馈。

### 核心界面设计详情
1. 登录与数据大屏仪表盘交互设计：账号密码输入框结合角色下拉单选。登录成功后，跳转至动态仪表盘首页。数据展示：使用 ECharts 渲染系统核心指标卡片（总注册人数、今日宣讲会数量）与签到趋势动态图表，提供直观的运营视图。
2. 宣讲会管理与预约大厅组件化布局：使用 Element-Plus 的 el-table 展示宣讲会列表，包含分页、多条件检索（日期、企业名）。操作反馈：学生点击“立即预约”按钮时，触发后端高并发接口，前端配置防抖机制及 v-loading 加载动画，成功后弹出消息提示框。
3. 人脸签到核验台多媒体交互：左侧为摄像头实时画面（video 标签），右侧为当前比对结果状态墙。业务逻辑：定时抓帧转换为 Base64 格式通过 Axios 传至后端，比对成功后，界面亮起绿灯提示“签到成功”，并自动更新签到统计面板。
